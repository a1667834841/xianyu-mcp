# 闲鱼搜索稳定性与 Sign 感知页面内 API 主链路设计

日期: 2026-04-07
状态: 待审阅

## 1. 背景

当前闲鱼搜索存在两类直接问题：

1. 某些关键词搜索结果不稳定，甚至出现明显应有结果却返回 0 条的情况。
2. 当 `rows > 30` 时，现有搜索链路容易超时或翻页不稳定，无法持续拿到目标数量。

现有实现的核心问题在于：

- 主流程依赖页面输入框、搜索按钮和“下一页”按钮等 DOM 交互。
- 响应采集依赖页面时序，容易受到旧响应残留、按钮点击失效、页面结构变化等因素影响。
- 闲鱼搜索接口带有 `sign` 校验，不能简单在 Python 服务端直接伪造请求并稳定复用。

用户明确要求：

- 同时解决关键词准确性和大于 30 条时的分页稳定性。
- 可以尝试更准确的新主链路，但原有逻辑必须保留。
- 若涉及 `sign`，必须考虑其依赖浏览器环境、cookie 和时间戳，不能做不可靠的服务端裸请求方案。

## 2. 目标

### 2.1 功能目标

- 保证关键词搜索链路比当前页面交互方式更稳定、更可验证。
- 在 `rows > 30` 时支持稳定翻页并持续累计唯一商品结果。
- 保留当前 `_BrowserSearchImpl` 页面交互搜索逻辑作为兜底路径。
- 在搜索结果中补充搜索元信息，说明本次使用的搜索引擎和降级原因。

### 2.2 约束

- 不能将“Python 侧直接构造 HTTP 请求并自行生成 sign”作为主方案。
- 新主链路必须运行在真实浏览器页面环境内，复用当前登录态、cookie、origin、referer 和页面上下文。
- 外部 MCP 工具调用方式保持不变。
- 修复应聚焦搜索稳定性，不扩展到无关功能。

### 2.3 非目标

- 本轮不实现完全脱离浏览器环境的独立闲鱼 API SDK。
- 本轮不重写登录、保活、发布等其他业务流程。
- 本轮不承诺彻底规避平台风控，只尽量采用更接近真实页面环境的低风险路径。

## 3. 关键结论

结合当前项目实现和参考实现，设计基于以下结论：

1. 闲鱼搜索接口的 `sign` 与 `token`、时间戳、`appKey`、请求体字符串直接相关。
2. 参考实现中的搜索请求虽然是直接 `fetch` 接口，但仍然依赖浏览器页面环境中的 cookie 和 `credentials: include`。
3. 因此，“页面内接口搜索”可作为更稳的主链路；“服务端裸请求搜索”不适合作为本轮方案。
4. 原有页面交互搜索仍有保留价值，应作为兼容兜底路径，而不是直接删除。

## 4. 推荐方案

推荐方案为：保留旧页面交互搜索，实现“页面内 API 搜索”为主链路，并增加统一搜索编排层。

整体结构如下：

```text
XianyuApp.search_with_meta
  -> SearchCoordinator
      -> StableSearchRunner
          -> PageApiSearchClient       # 主链路
          -> _BrowserSearchImpl        # 兜底链路
```

核心原则：

- 主链路在页面上下文内发起搜索请求，由页面脚本负责使用当前会话环境生成请求及 `sign`。
- Python 侧只负责控制搜索流程、去重、超时、停止条件、诊断信息和降级策略。
- 原有 DOM 搜索逻辑不删除，仅在新主链路失败时使用。

## 5. 架构设计

### 5.1 SearchCoordinator

`SearchCoordinator` 负责搜索路径调度。

职责：

- 接收 `SearchParams`
- 优先调用 `StableSearchRunner` + `PageApiSearchClient`
- 发现主链路不可用时切换到 `_BrowserSearchImpl`
- 汇总 `SearchOutcome`

它不直接处理浏览器 DOM，不直接构造接口签名，只负责协调。

### 5.2 PageApiSearchClient

`PageApiSearchClient` 是本轮新增主链路边界。

职责：

- 在 Playwright 控制的页面上下文中执行搜索脚本
- 由页面脚本在浏览器环境内读取 cookie、构造请求参数、生成 `sign`
- 发起搜索接口请求并返回结构化分页数据

边界要求：

- `sign` 的计算和请求发起都在页面环境内完成
- Python 侧不保存独立签名算法副本作为主路径依赖
- 页面脚本的输入输出保持最小化：输入查询参数，输出当前页商品列表和分页元数据

### 5.3 StableSearchRunner

`StableSearchRunner` 负责稳定搜索流程。

职责：

- 固定一次搜索会话内的查询快照
- 控制分页推进
- 以 `item_id` 去重
- 统计 `stale_pages`
- 达到 `rows` 后立即截断返回
- 生成 `engine_used`、`fallback_reason`、`pages_fetched` 等元信息

该层不关心页面 DOM，只关心“拿到一页结构化结果后如何持续收敛到最终答案”。

### 5.4 _BrowserSearchImpl

保留现有 `_BrowserSearchImpl`。

角色调整为：

- 不再作为首选主路径
- 作为主链路失败时的兼容兜底实现
- 继续支持现有页面交互搜索方式，避免页面内 API 路径临时失效时整条搜索能力不可用

## 6. 数据流设计

### 6.1 主链路搜索流程

推荐的主链路步骤如下：

1. 复用当前 `work_page`，确保浏览器和登录态可用。
2. 在页面上下文中注入或调用搜索辅助脚本。
3. 由页面脚本根据当前 cookie、时间戳、`appKey` 和请求体生成 `sign`。
4. 页面脚本直接发起搜索接口请求，返回本页结果。
5. `StableSearchRunner` 收到本页结果后按 `item_id` 去重并累计。
6. 若未达到 `rows`，仅推进分页参数，继续请求下一页。
7. 达到目标数量或命中停止条件后，返回 `SearchOutcome`。

### 6.2 查询快照冻结

为避免翻页过程中请求语义漂移，一次搜索会话中应冻结以下字段：

- `keyword`
- `min_price`
- `max_price`
- `free_ship`
- `sort_field`
- `sort_order`
- 页面内 API 所需的其他过滤参数

翻页时只允许变更分页相关参数，不允许悄悄改写关键词或排序条件。

### 6.3 关键词一致性校验

新主链路必须具备关键词一致性校验。

最低要求：

- 页面内 API 请求前后，系统必须能够确认真正发出的关键词与用户请求关键词一致。
- 如果发现页面脚本最终使用的查询词与目标关键词不一致，应判定主链路失败并触发降级，而不是返回“看起来有结果但关键词错误”的数据。

## 7. 停止条件与结果元信息

### 7.1 停止条件

保留并扩展停止条件：

- `target_reached`: 已达到请求的目标数量
- `stale_limit`: 连续多页没有新增唯一商品
- `api_unavailable_fallback`: 页面内 API 主链路不可用，已切到兜底
- `browser_fallback_exhausted`: 兜底链路执行完但仍未达到目标
- `error`: 两条链路都失败或出现未恢复错误

### 7.2 搜索元信息

`SearchOutcome` 及 MCP 输出应在保留兼容字段的基础上，新增以下信息：

- `engine_used`: `page_api` 或 `browser_fallback`
- `fallback_reason`: 仅在发生降级时填写
- `pages_fetched`: 实际抓取页数

兼容字段继续保留：

- `requested`
- `total`
- `stop_reason`
- `stale_pages`
- `items`

## 8. 降级策略

以下情况必须自动从页面内 API 主链路降级到 `_BrowserSearchImpl`：

- 页面脚本初始化失败
- 页面内接口请求超时
- 返回结构不符合预期
- 关键词一致性校验失败
- 检测到登录态异常或疑似风控拦截

降级策略要求：

- 降级必须自动发生，不要求用户重新调用搜索
- 降级原因必须进入元信息和日志
- 如果兜底链路成功，仍然返回成功结果，但要标记 `engine_used=browser_fallback`

## 9. 测试设计

### 9.1 页面内 API 客户端单测

新增测试覆盖：

- 页面脚本返回多页结果时，能够累计到 `rows=50/100`
- 第二页开始无新增商品时，正确触发 `stale_limit`
- 页面脚本返回的关键词与请求关键词不一致时，主链路失败
- 页面脚本超时或抛异常时，调用方能得到可降级错误

### 9.2 编排层单测

新增测试覆盖：

- 主链路成功时，不调用旧链路
- 主链路失败时，自动切到 `_BrowserSearchImpl`
- 返回中包含 `engine_used` 和 `fallback_reason`
- 旧链路成功时，对外结构保持兼容

### 9.3 分页回归测试

在现有 `tests/test_search_pagination.py` 基础上补充：

- 新主链路在 `rows > 30` 时可稳定累计结果
- 空结果时不会误复用旧响应
- 多页去重和 stale page 计算保持正确

### 9.4 集成层测试

`mcp_server/http_server.py` 和 `mcp_server/server.py` 需要验证：

- 原有返回字段保持兼容
- 可新增 `engine_used`、`fallback_reason`
- 当主链路失败但 fallback 成功时，返回仍然是成功响应

## 10. 验收标准

本轮完成标准为：

1. `泡泡玛特` 这类原先不稳定的关键词，不再经常出现“应有结果却返回 0 条”的问题。
2. `rows > 30` 时，优先通过页面内 API 主链路稳定翻页获取结果。
3. 页面内 API 主链路失败时，系统自动回退到原页面交互搜索逻辑。
4. MCP 工具调用方式不变。
5. 自动化测试至少覆盖关键词准确性、多页累计、stale limit 和 fallback。

## 11. 风险与取舍

### 11.1 风险

- 页面内脚本仍然依赖闲鱼接口参数和返回结构，若平台改版仍可能受影响。
- 即使运行在真实浏览器环境中，也不能完全消除平台风控风险。
- 旧链路和新链路并存，会增加一定维护成本。

### 11.2 取舍

- 本设计优先“更稳定、更可验证”，而不是最少代码改动。
- 本设计保留旧逻辑，接受一定复杂度增加，以换取更稳的主路径和明确的回退空间。
- 本设计不将签名算法硬编码为 Python 服务端主逻辑，避免未来与页面实现长期漂移。

## 12. 实施边界

本轮实现应聚焦以下文件和模块：

- `src/core.py`: 搜索编排、结果元信息、fallback 协调
- 新增页面内 API 搜索模块，例如 `src/search_api.py` 或同类模块
- `tests/test_search_pagination.py`: 分页与去重回归测试
- 新增页面内 API/协调层测试
- `mcp_server/http_server.py` 与 `mcp_server/server.py`: 输出新增元信息

本轮不应扩展到以下范围：

- 发布能力重构
- 登录重构
- 保活机制重构
- 完整独立 SDK 化改造
