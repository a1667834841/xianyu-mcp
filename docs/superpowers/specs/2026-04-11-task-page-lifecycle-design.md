# 任务页生命周期简化设计

日期：2026-04-11

## 背景

当前项目已经引入了按角色缓存页面的思路，例如 `search_page`、`session_page`、`publish_page`、`keepalive_page`。这比单一 `work_page` 更进一步，但仍然存在两个实际问题：

- 页面数量容易随着任务类型和异常路径变复杂，不利于长期控制
- 页面长期复用后容易残留导航状态、弹窗、监听器或中间 DOM 状态，下一次任务复用时需要额外清理

结合当前业务目标，核心诉求不是做更复杂的页面池，而是让页面总量稳定、异常恢复简单，并尽量减少“脏页面复用”的问题。

## 目标

本次设计的目标如下：

- 让浏览器中的业务页面数量保持很少且可预测
- 避免大多数任务之间复用脏页面状态
- 让任务在成功和异常两种路径下都能可靠回收页面
- 保留登录扫码链路和后台保活链路所需的长生命周期页面
- 在现有 Playwright + CDP 架构上做最小必要改动

## 非目标

本次不做以下事情：

- 不改为多 `BrowserContext` 架构
- 不引入多浏览器实例
- 不重写搜索、发布、登录的核心业务流程
- 不引入复杂页面池、优先级队列或空闲淘汰器
- 不改变对外 MCP 工具接口与参数结构

## 方案选型

### 方案 A：固定角色页长期复用

为搜索、会话、发布分别保留长期存在的角色页。

优点：

- 语义清晰
- 调试直观

缺点：

- 页面数偏多
- 长期复用后更容易积累脏状态
- 需要更多清理和占用管理规则

### 方案 B：双固定页加临时搜索页

保留 `main_page` 和 `publish_page`，冲突时为搜索创建唯一临时页。

优点：

- 页面数较少
- 比全固定角色页更节制

缺点：

- 仍然需要管理复用优先级和借页策略
- `main_page` 仍会同时承担多类业务，状态边界不够干净

### 方案 C：短生命周期任务页 + 少量保留页（推荐）

除登录扫码和保活外，其余业务任务都使用一次性页面，任务结束或异常时立刻关闭。

优点：

- 页面数天然受控
- 异常恢复最简单
- 页面状态污染最少
- 最符合“不要开太多页面”的目标

缺点：

- 每次任务都需要重新创建并导航页面，性能略差于长期复用

结论：采用方案 C。

## 页面模型

### 页面类型

本次设计只保留三类页面：

- `keepalive_page`
  - 常驻页面
  - 仅供后台保活服务使用
- `session_page`
  - 登录链路专用页面
  - 仅在 `show_qr` / `login` 期间保留
  - 登录完成、失败、超时或取消后关闭
- `task_page`
  - 一次性任务页
  - 用于 `search`、`check_session`、`refresh_token`、`get_detail`、`publish`
  - 每次任务开始创建，结束或异常时关闭

### 页面数量上限

常见情况下页面数如下：

- 空闲时：仅 `keepalive_page`
- 登录中：`keepalive_page + session_page`
- 执行业务任务时：`keepalive_page + task_page`
- 登录中同时执行业务任务时：`keepalive_page + session_page + task_page`

因此业务相关页面的上限保持在一个非常可控的范围内，不再随任务类型长期累积。

## 生命周期规则

### `keepalive_page`

- 懒创建
- 一旦创建即长期保留
- 仅在浏览器关闭、连接重建或保活服务停止时销毁

### `session_page`

- 仅在二维码展示和登录会话期间创建并保留
- 如果页面关闭、崩溃或脱离 `context.pages`，则按需重建
- 当登录结束、失败、超时或明确取消时立即关闭

### `task_page`

- 每次业务任务单独创建
- 业务代码必须通过统一租约拿到页面，不允许直接散落 `new_page()`
- 无论任务成功还是异常，都在 `finally` 中关闭
- 不保留到下一次任务，不做长期缓存

## 调度与协调层设计

### 核心原则

不构建重型页面池，只补一层很薄的 `PageCoordinator / PageLease`，把页面调度规则集中起来。

### `AsyncChromeManager` 的职责

- 负责 Playwright / CDP 连接
- 维护 `browser` 与 `context` 生命周期
- 提供底层页面能力：创建页、关闭页、检测页是否仍可用

### `PageCoordinator` 的职责

- 维护 `_keepalive_page`
- 维护 `_session_page`
- 提供一次性 `task_page` 的租约申请
- 统一检查页面是否失效并在必要时重建
- 控制短生命周期业务任务的并发上限

`PageCoordinator` 不负责业务逻辑本身，也不负责复杂调度策略，只负责“页面怎么拿、什么时候关”。

### `PageLease` 的职责

提供一个很轻的租约对象，包含：

- `page`
- 释放逻辑
- 是否为临时页面的自动回收语义

行为规则：

- `temporary=True` 的租约在退出时自动关闭页面
- `temporary=False` 的租约只释放占用，不关闭页面

这样业务模块只需要使用统一的 lease 接口，而不再自己判断页面是否要回收。

## 并发规则

### 默认策略

为避免短时间内再次出现过多业务页，本次设计默认只允许一个 `task_page` 同时存在。

这意味着：

- `search`、`check_session`、`refresh_token`、`get_detail`、`publish` 默认串行执行
- `session_page` 不占用 `task_page` 并发配额
- `keepalive_page` 完全独立，不参与业务页竞争

### 选择这个策略的原因

- 任务页设计的核心目标是控制页面总量，而不是提升并发度
- 当前用户优先级是“页面不要太多”，不是“所有业务并行”
- 串行短任务加一次性页面可以显著简化调度和异常恢复

如果未来确认需要提高吞吐量，可以在不改变接口的前提下，将 `task_page` 并发上限从 1 调整为 2，但这不属于本次范围。

## 模块落地方式

### `src/browser.py`

- 保留底层浏览器连接与页面创建能力
- 弱化现有固定业务角色页缓存的主路径地位
- 底层仍可保留 `get_keepalive_page()` 一类方法，但业务侧不再依赖散落的角色页缓存

### 新增页面协调模块

建议新增一个轻量模块，例如 `src/page_coordinator.py`，用于承载：

- `PageCoordinator`
- `PageLease`

引入新模块的原因是让页面生命周期规则集中，而不是继续分散在 `browser.py`、`core.py`、`session.py` 中。

### `src/core.py`

将以下业务入口改为通过 `PageCoordinator.lease_task_page()` 运行：

- `search`
- `search_with_meta`
- `publish`
- `get_detail`

这些任务不再假设存在长期保留的业务页。

### `src/session.py`

分成两类路径：

- `show_qr` / `login`
  - 使用 `session_page`
  - 登录链路结束后关闭 `session_page`
- `check_session` / `refresh_token`
  - 使用一次性 `task_page`
  - 执行完成后关闭

这样可以把“需要用户交互保留页面”和“纯后台任务页面”明确分开。

### `src/keepalive.py`

- 继续只使用 `keepalive_page`
- 不接入 `task_page` 并发控制
- 不与登录或业务任务共用页面

## 错误处理与恢复

### 页面有效性检查

在发放租约前，协调层需要统一检查：

- 页面是否为 `None`
- 页面是否 `is_closed()`
- 页面是否仍存在于 `context.pages`

若不满足条件则重建页面。

### 任务执行中的异常

- `task_page` 出现异常时，当前任务返回错误
- 无论异常来自业务逻辑还是页面操作，都必须在 `finally` 中关闭页面
- 后续任务重新申请新页，不依赖失败页面恢复

### 登录页面异常

- 若 `session_page` 在登录链路中关闭或崩溃，则下次登录相关调用时重建
- 登录结束后主动关闭，避免“扫码页一直挂着”

### 不做的事情

- 不做自动多次重试
- 不做隐藏式失败恢复
- 不把页面问题和业务问题混在一起吞掉

这样可以保持错误来源清晰，减少误判。

## 兼容性要求

对外接口保持不变，包括但不限于：

- `xianyu_login`
- `xianyu_show_qr`
- `xianyu_check_session`
- `xianyu_refresh_token`
- `xianyu_search`
- `xianyu_publish`

本次只调整内部页面生命周期和调度方式，不改动 MCP 工具接口、参数和返回结构。

## 测试设计

### 单元测试

需要新增或调整以下测试：

1. `task_page` 生命周期
- 获取租约时创建新页
- 成功退出时自动关闭
- 异常退出时也自动关闭

2. `session_page` 生命周期
- 登录链路期间复用同一 `session_page`
- 登录结束或失败后关闭
- 页面失效后可重建

3. `keepalive_page` 生命周期
- 懒创建并长期复用
- 不受短任务关闭影响

4. 并发控制
- 同时发起两个短任务时，第二个任务等待 `task_page` 配额
- 不允许因为并发短任务而不断开新页

5. 页面失效恢复
- 页面被手动关闭后，再次申请时自动获取新页

### 集成测试

至少覆盖以下场景：

1. 登录期间执行搜索
- `session_page` 保留
- 搜索使用独立 `task_page`
- 搜索结束后 `task_page` 关闭

2. 连续执行多个搜索或会话检查
- 每次任务使用新页面
- 前一个任务页面关闭后不会残留到下一次

3. 发布任务异常
- 页面在异常路径下也被关闭

4. 保活与业务任务并存
- `keepalive_page` 持续存在
- `task_page` 的创建和销毁不影响保活

## 风险与缓解

### 风险 1：频繁开关页面带来性能损耗

缓解：

- 当前优先级是稳定和简化，不是极限吞吐量
- 后续若确认成为瓶颈，可在协调层内提高短任务页并发或引入受控复用

### 风险 2：旧代码仍直接调用 `context.new_page()`

缓解：

- 将业务页面申请统一收敛到 `PageCoordinator`
- 在关键路径测试中覆盖“任务结束必须关页”的行为

### 风险 3：登录状态与临时任务页边界不清

缓解：

- 明确只有登录链路保留 `session_page`
- `check_session` / `refresh_token` 明确归类为短任务页

## 结论

本次设计采用“短生命周期任务页 + 少量保留页”的方式，具体为：

- `keepalive_page` 常驻
- `session_page` 只在登录期间保留
- 其余业务全部使用一次性 `task_page`

在实现上，仅新增一层很薄的 `PageCoordinator / PageLease` 来统一页面申请和回收，不引入复杂页面池。这一方案最符合当前项目对页面数量控制、异常恢复简单以及最小改动的要求。
