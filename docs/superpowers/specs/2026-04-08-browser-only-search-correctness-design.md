# 闲鱼 Browser-Only 搜索正确性优先设计

日期: 2026-04-08
状态: 待审阅

## 1. 背景

当前闲鱼搜索存在两个核心问题：

1. 关键词正确性不稳定。搜索 `泡泡玛特` 时，实际可能返回 `Mac mini` 一类明显无关结果。
2. 当 `rows > 30` 时，分页抓取容易混入旧响应、翻页失效或提前停住，无法稳定拿到目标数量。

此前系统尝试过“页面内 API 主链路 + 浏览器 fallback”的方案，但当前目标已经收敛：

- 先把搜索结果做对，再考虑性能。
- 只保留“操作页面 + 监听真实接口”这一条主链路。
- 如果强校验失败，直接报错，不返回疑似错误结果。

## 2. 目标

### 2.1 功能目标

- 搜索 `泡泡玛特` 时，结果必须与目标关键词语义相关，不能再出现明显错误品类冒充成功。
- 支持 `rows=100` 的分页抓取。
- 结果必须按 `item_id` 去重。
- 搜索失败时返回明确错误原因，而不是用错误结果充当成功。

### 2.2 范围

本轮只保证以下主线能力：

- `keyword`
- 多页抓取直到 `rows=100`
- 去重
- 强校验

### 2.3 非目标

本轮不保证以下能力的稳定性：

- `min_price`
- `max_price`
- `free_ship`
- `sort_field`
- `sort_order`

本轮也不继续增强以下方向：

- PageApi 作为主查询路径
- 服务端直接构造带 sign 的搜索请求
- 为了成功率而返回语义错误但“看起来有数据”的结果

## 3. 设计原则

本轮设计遵循四条原则：

1. 正确性优先于性能。
2. 一条主链路优先于多链路复杂协调。
3. 强校验失败直接报错，不做“将错就错”的降级返回。
4. 只做当前验收必须的最小系统，不扩展无关功能。

## 4. 推荐方案

推荐方案为：Browser-Only 搜索主链路。

整体结构如下：

```text
XianyuApp.search_with_meta
  -> BrowserSearchRunner
      -> SearchSession
      -> ResponseCapture
      -> ResultValidator
```

核心思路：

- 每次搜索都从闲鱼首页开始。
- 通过首页搜索框输入关键词并回车。
- 只监听这次用户动作触发的真实搜索接口响应。
- 第 1 页必须通过强校验，失败立即报错。
- 翻页后继续监听新响应并按 `item_id` 去重累计，直到拿满 `100` 条或命中停止条件。

## 5. 架构设计

### 5.1 XianyuApp.search_with_meta

职责：

- 接收外部搜索参数。
- 创建并调用 `BrowserSearchRunner`。
- 返回 `SearchOutcome`。

边界：

- 不再优先走 `PageApiSearchClient`。
- 不再自动切换到“错误结果也算成功”的 fallback 逻辑。

### 5.2 BrowserSearchRunner

职责：

- 编排完整搜索流程。
- 控制搜索起点、首次搜索、翻页、去重、停止条件和最终结果输出。

边界：

- 它是对外唯一搜索执行入口。
- 它不负责平台级业务推断，只负责一次搜索流程的稳定推进。

### 5.3 SearchSession

职责：

- 保存一次搜索会话的运行状态。
- 隔离“本次搜索”与“旧页面残留状态”。

建议字段：

- `keyword`
- `started_at`
- `captured_responses`
- `seen_item_ids`
- `items`
- `pages_fetched`
- `stale_pages`

### 5.4 ResponseCapture

职责：

- 监听浏览器响应。
- 过滤掉与本次搜索无关的响应。
- 为第一页和后续翻页提供“新响应到达”的判断能力。

边界：

- 只接收真实搜索接口响应。
- 不接收 `shade`、`suggest`、预取响应和触发点之前的旧响应。

### 5.5 ResultValidator

职责：

- 对第一页结果执行强校验。
- 拦截明显语义跑偏的结果。

边界：

- 第一页必须通过。
- 失败立即返回 `validation_failed`，不继续翻页。

## 6. 一次搜索的数据流

### 6.1 建立搜索会话

开始搜索时创建新的 `SearchSession`，并重置以下内容：

- 本次响应缓存
- 已见 `item_id`
- 已累计商品结果
- 页码计数
- stale page 计数
- 本次错误状态

目标是确保本次搜索拥有明确边界，不复用上一次搜索的残留状态。

### 6.2 回到可控起点

每次搜索都必须先回到闲鱼首页，并尝试完成以下准备动作：

1. 导航到首页。
2. 关闭可能遮挡搜索框的登录弹窗或蒙层。
3. 确认首页搜索框可交互。

如果搜索框不可用或被持续拦截，则直接返回 `interaction_failed`。

### 6.3 触发第一页搜索

第一页搜索必须通过首页搜索框完成：

1. 点击搜索框。
2. 清空旧内容。
3. 输入目标关键词。
4. 回车触发搜索。

从触发搜索的时间点开始，系统只接受该时间点之后到达的真实搜索接口响应。

### 6.4 捕获第一页响应

第一页只接受满足以下条件的响应：

- 属于真实搜索接口。
- 在本次搜索触发时间之后到达。
- 能成功解析出商品列表。

如果在超时时间内没有拿到合法第一页响应，则返回 `response_timeout`。

### 6.5 第一页强校验

第一页必须通过双层强校验：

1. 硬一致性校验
2. 结果语义校验

硬一致性校验要求：

- 本次会话目标关键词必须等于实际触发搜索时使用的关键词。

结果语义校验要求：

- 只做“明显跑偏拦截”。
- 如果前几条商品标题明显属于错误品类，则直接判定失败。

例如搜索 `泡泡玛特` 时，如果第一页前几条主要是 `Mac mini`、电脑主机或相关数码品类，则必须返回 `validation_failed`。

### 6.6 翻页与累计

只有第一页强校验通过后，才允许继续翻页。

每次翻页流程如下：

1. 记录翻页前已捕获响应数量。
2. 通过页面真实翻页动作进入下一页。
3. 等待翻页后新增的搜索接口响应。
4. 解析出商品列表。
5. 按 `item_id` 去重后累计。

后续页不再重复执行第一页那种严格语义校验，但必须保证：

- 仍属于同一次搜索会话。
- 不混入旧响应。
- 不重复累计相同 `item_id`。

## 7. 停止条件

本轮只保留明确、可解释的停止条件：

- `target_reached`: 已达到请求的 100 条目标数量。
- `stale_limit`: 连续若干页没有新增唯一商品。
- `validation_failed`: 第一页强校验失败。
- `interaction_failed`: 页面交互失败，例如搜索框不可用或被弹窗持续拦截。
- `response_timeout`: 等不到合法新响应。
- `error`: 未归类异常。

系统必须真实返回停止原因，不能把失败包装成成功。

## 8. 返回结构

继续复用现有 `SearchOutcome` 结构，但调整字段语义：

- `engine_used`: 固定为 `browser_interaction`
- `fallback_reason`: 本轮可为空，不再作为主逻辑的一部分
- `stop_reason`: 必须反映真实停止原因
- `requested_rows`: 用户请求数量
- `returned_rows`: 实际返回数量
- `pages_fetched`: 实际抓取页数
- `stale_pages`: 连续无新增页数
- `items`: 去重后的商品列表

如果搜索失败，允许 `returned_rows < requested_rows` 或直接为空，但不得返回明显错误语义的数据充当成功结果。

## 9. 错误处理策略

错误处理统一遵循“明确失败，不伪装成功”：

- 搜索框找不到或不可交互：`interaction_failed`
- 弹窗、蒙层导致点击或回车持续失败：`interaction_failed`
- 等不到第一页响应：`response_timeout`
- 第 1 页结果明显跑偏：`validation_failed`
- 翻页后连续无新增结果：`stale_limit`
- 解析异常或未知异常：`error`

本轮不再提供“强校验失败后自动重试并返回当前最好结果”的路径。

## 10. 测试设计

### 10.1 单元测试

新增或调整测试覆盖：

- `ResponseCapture` 只接收真实搜索接口响应。
- `ResponseCapture` 不接收 `shade`、`suggest` 和旧响应。
- 去重逻辑按 `item_id` 正常工作。
- `stale_pages` 统计正确。
- `ResultValidator` 能拦截明显跑偏结果。

### 10.2 集成测试

重点验证以下场景：

- 搜索 `泡泡玛特` 且 `rows=100`。
- 返回结果不重复。
- 前若干条标题与 `泡泡玛特` 语义相关。
- 不再出现 `Mac mini` 这类明显错误品类冒充成功。
- 强校验失败时，返回错误而不是错误结果。

### 10.3 验收标准

本轮完成标准为：

1. `泡泡玛特` 搜索不再出现明显错误品类冒充成功。
2. `rows=100` 可以通过多页抓取持续累计结果。
3. 结果按 `item_id` 去重。
4. 第 1 页强校验失败时直接报错。
5. 系统返回的 `stop_reason` 能真实解释本次搜索为何结束。

## 11. 风险与取舍

### 11.1 风险

- 页面结构变化可能影响首页搜索框定位或翻页动作。
- 登录弹窗、蒙层和风控页面仍可能中断真实交互。
- “语义相关”校验本质上只能先做明显跑偏拦截，不能在本轮覆盖全部边界语义情况。

### 11.2 取舍

- 接受性能一般，优先换取正确性。
- 接受这轮只做主线，不把价格、包邮、排序一起做稳。
- 接受放弃复杂 fallback，换来更清晰的结果语义。

## 12. 实施边界

本轮实现应聚焦以下位置：

- `src/core.py`
- 如有必要，可拆出轻量搜索会话/响应过滤/校验模块
- `tests/test_search_pagination.py`
- 与搜索结果结构相关的单测

本轮不应扩展到以下范围：

- 发布能力
- 登录重构
- Cookie 保活机制重构
- PageApi 方案继续增强
- 搜索过滤排序的完整稳定化
